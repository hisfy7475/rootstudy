# CAPS 출입기록 연동 이야기

## 시작

"기존에 쓰던 앱에서 잘 되고 있으니까 API 있을 거예요"라는 말을 믿고 시작했다.

막상 들여다보니 그런 게 아니었다. CAPS 출입관리 시스템은 어딘가의 **내부 컴퓨터에 MSSQL로 출입 기록을 쌓고 있었고** (MySQL이 아니라 Microsoft SQL Server다), 다른 앱이 어떻게 가져다 쓰는지는 알 수 없었다. 공개 API 같은 건 없었다.

결국 그 MSSQL에 직접 붙어서 데이터를 읽어오는 수밖에 없었다.

---

## 문제: Vercel은 항시 떠있는 서버가 아니다

실시간으로 CAPS DB에서 출입 기록을 가져오려면 서버가 항상 연결을 유지하거나, 이벤트가 생길 때마다 DB를 조회해야 한다.

Vercel은 요청이 들어올 때만 함수가 실행되는 구조라, Railway 같은 항시 실행 서버 없이 이걸 진짜 실시간으로 처리하는 건 어렵다. 비용도 올라간다.

---

## 타협: 1분 크론잡

클라이언트에게 물어봤다.

> "완전 실시간이 필요하세요? 그러면 별도 서버 비용이 발생해요."

1분 정도 딜레이면 충분하다고 했다. 그래서 **Vercel Cron을 1분 주기로 돌려서 CAPS MSSQL → Supabase로 동기화하는 방식**으로 구현했다.

```
매 1분
CAPS MSSQL (내부 서버) → caps-sync 크론잡 → Supabase PostgreSQL
                                                      ↓
                                               웹/앱에서 조회
```

서버 추가 없이 Vercel 안에서 해결됐다.

---

## 다음에 비슷한 상황이 생긴다면

> **"API 있을 거예요"는 사실 확인이 필요하다.** 레거시 시스템일수록 내부 DB 직접 접근이 전부인 경우가 많다.

비슷한 상황에서 쓸 수 있는 옵션들:

| 상황 | 방법 |
| --- | --- |
| 딜레이 허용 (현재처럼) | **크론잡 폴링** — 간단하고 추가 인프라 없음 |
| 진짜 실시간 필요 | **Railway/Render 등 항시 실행 서버** 별도 운영 |
| 내부망 접근이 막혀있을 때 | 고객사 측에서 **웹훅 or API 직접 제공**받는 게 근본 해결 |

지금 구조의 가장 취약한 부분은 **CAPS DB 서버가 내부망이라 Vercel에서 직접 붙는 것 자체가 가능한지 여부가 환경에 따라 달라진다**는 점이다. 이번엔 됐지만, 고객사 네트워크 정책이 바뀌거나 방화벽이 강화되면 연결이 끊길 수 있다.
