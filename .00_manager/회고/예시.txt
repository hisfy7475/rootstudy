# 메티스 프로젝트 기술스택 회고

## 1. 프로젝트 전체 구조 — 모노레포

**Turborepo + npm workspaces**

전체 코드베이스를 하나의 Git 리포지토리에서 관리하는 모노레포 방식을 채택했다.

```
metis/
├── apps/
│   ├── web/       ← 관리자·운영 웹앱 (Next.js)
│   ├── public/    ← 공개 랜딩·통계 페이지 (Next.js 경량)
│   └── mobile/    ← iOS·Android 앱 (Expo)
└── packages/
    ├── ui/               ← 웹/앱 공용 컴포넌트
    ├── typescript-config/ ← 공유 tsconfig
    └── eslint-config/    ← 공유 ESLint 설정
```

---

## 2. 프론트엔드 — Next.js + React Native (Expo)

### 웹 — Next.js

메인 웹과 공개 페이지 모두 **Next.js** 기반. React 서버 컴포넌트, API Route를 함께 활용한다.

- `apps/web`: 메인 서비스 웹. 웹뷰 콘텐츠 + 어드민 기능 포함
- `apps/public`: 외부에 공개되는 랜딩·통계 페이지. 별도 Next.js 앱으로 분리

스타일링은 **Tailwind CSS v4** 공통 사용.

### 앱 — React Native (Expo)

iOS · Android 모두 **Expo** 기반으로 개발하고 App Store · Play Store에 출시했다.

- 핵심 화면(홈 피드, 딜 상세)은 네이티브로 구현
- 나머지 화면은 `apps/web`을 WebView로 임베드하는 하이브리드 구조

스타일링은 **NativeWind** (Tailwind를 React Native에서 사용)로 웹과 동일한 클래스 체계 유지.

---

## 3. 데이터베이스 & 인증 — Supabase

DB, 인증, 파일 스토리지를 모두 **Supabase**로 처리했다.

- **PostgreSQL**: 메인 데이터베이스
- **Auth**: 이메일 + 소셜 로그인 (Apple, Google, 카카오, 네이버)
- **Storage**: 이미지·영상 파일

---

## 4. 백엔드 — Vercel에 올라간 Next.js가 담당

별도의 백엔드 서버가 없다. **`apps/web`의 Next.js API Routes가 백엔드 역할을 겸한다.**

처음에 웹뷰앱으로 시작했기 때문에 자연스럽게 이 구조가 됐는데, 결제 처리, 이메일 발송, 크론 작업, 모바일 세션 연동 등 서버 로직이 전부 Vercel에 배포된 Next.js 안에 들어가 있다.

서비스가 커질수록 **백엔드를 별도로 분리했어야 한다**는 아쉬움이 있다.

---

## 5. 빌드 & 배포

로컬에서 빌드하지 않고 전부 클라우드 빌드로만 진행했다.

| 대상 | 도구 |
| --- | --- |
| iOS · Android 앱 빌드 | **EAS Build** (Expo 클라우드 빌드) |
| 앱 OTA 업데이트 | **EAS Update** (스토어 심사 없이 JS 번들 배포) |
| 웹 배포 | **Vercel** (push하면 자동 배포) |
| 정기 작업 (크론) | **Vercel Cron** |

# 웹뷰앱으로 가다가 네이티브 앱이 되어버린 이야기

---

## 1단계: 웹뷰앱 추천

초기 기획은 "단순 쇼핑몰"이었다. 복잡한 인터랙션이 없을 것 같았고, 개발 속도와 유지보수를 고려했을 때 웹뷰앱이 합리적인 선택이었다.

- 웹 하나만 만들면 앱도 커버됨
- 웹·앱 코드베이스를 분리할 필요 없음
- 스토어 배포는 웹뷰를 감싼 쉘 앱으로 처리

---

## 2단계: 아이디어가 앱 중심으로 바뀌기 시작

개발이 진행되면서 기획이 달라졌다. 단순 쇼핑몰이 아니라 **틱톡 스타일의 세로 스크롤 피드**, 스와이프 제스처, 영상 재생 등 앱스러운 인터랙션이 핵심 경험으로 자리 잡기 시작했다.

웹뷰에서 이걸 구현했더니 퍼포먼스가 눈에 띄게 아쉬웠다.

- 세로 스크롤 피드에서 프레임 드롭
- 스와이프 제스처 반응 속도가 네이티브 대비 뭉툭함
- 영상 전환 시 버벅임

사용자가 가장 많이 머무는, 가장 중요한 화면에서 경험이 나빴다.

---

## 3단계: 부분 네이티브 전환 — 하이브리드로 타협

전체를 네이티브로 다시 짜는 건 일정상 불가능했다. 결론은 **핵심 화면만 네이티브, 나머지는 웹뷰 유지**하는 하이브리드 구조였다.

```
앱 탭 구조
├── 홈 (피드)     → 네이티브  (NativeTikTokFeed, DealSlide)
├── 딜 상세       → 네이티브  (NativeDealDetail)
└── 나머지 탭들   → WebView  (apps/web URL을 그대로 임베드)
```

퍼포먼스 문제는 해결됐다. 그러나 새로운 문제가 생겼다.

---

## 4단계: Auth가 꼬이기 시작

웹뷰와 네이티브가 공존하면서 **인증 상태를 두 곳에서 따로 관리**하게 됐다.

- **네이티브 영역**: Supabase JS 클라이언트가 `AsyncStorage`에 토큰 저장
- **웹뷰 영역**: 브라우저 쿠키 / `localStorage`에 토큰 저장

이 둘은 서로를 모른다. 그러다 보니 아래 상황들에서 문제가 터졌다.

**앱을 껐다 켤 때**
네이티브 쪽 토큰은 살아있는데 웹뷰 쪽 세션이 만료되어, 네이티브에서는 로그인 상태이지만 웹뷰 탭을 열면 로그인 화면이 뜨는 현상.

**로그아웃할 때**
네이티브에서 로그아웃 처리 → AsyncStorage 토큰 삭제. 그러나 웹뷰의 쿠키는 그대로 남아있어 웹뷰 탭에서는 여전히 로그인 상태.

**토큰 만료 타이밍이 다를 때**
Supabase 토큰은 자동 갱신되는데, 네이티브·웹뷰 각자가 독립적으로 refresh를 시도하다 보니 타이밍이 어긋나면 둘 중 하나가 401을 받고 로그아웃 처리되는 경우가 발생.

---

## 근본 원인

하이브리드 구조에서 Auth는 **단일 진실의 원천(Single Source of Truth)** 이 없었다.

```
네이티브 (AsyncStorage)  ←─── 서로 모름 ───→  웹뷰 (Cookie / localStorage)
         ↕                                              ↕
   Supabase 클라이언트 A                      Supabase 클라이언트 B
```

두 개의 Supabase 클라이언트 인스턴스가 같은 사용자의 세션을 각자 관리하는 상황이 됐다.

---

## 해결 방향 (또는 처음부터 설계했다면)

**실제로 쓴 방법 — SessionBridge**

네이티브에서 웹뷰를 로드할 때 토큰을 URL 파라미터나 `postMessage`로 넘겨서 웹뷰가 해당 토큰으로 세션을 초기화하도록 처리. `apps/mobile/components/SessionBridge`가 이 역할을 담당.

**처음부터 설계했다면**

네이티브를 전면에 내세울 것이 예상됐다면 처음부터 전체 네이티브로 설계하거나, 웹뷰 비중을 최소화하고 Auth는 네이티브 단일 관리 + 웹뷰는 stateless(매번 토큰 주입)로 설계하는 것이 깔끔했을 것이다.

---

## 교훈

> 초기 기획이 바뀔 가능성을 열어두고, 인터랙션이 핵심인 서비스라면
> 
> 
> 처음부터 네이티브 기반으로 잡는 것이 낫다.
> 
> 웹뷰에서 네이티브로의 전환 비용은, 처음부터 네이티브로 만드는 비용보다 훨씬 크다.
> 
> 특히 **Auth처럼 앱 전체에 걸쳐있는 레이어는 나중에 고치기가 매우 어렵다.**
>

# OTA 배포???

---

## 배경: 네이티브로 전환하고 나서 생긴 문제

웹뷰 영역은 Vercel에 배포하면 끝이라 수정사항을 바로 반영할 수 있었다. 문제는 네이티브 영역이었다. 홈 피드나 딜 상세처럼 네이티브로 짠 화면에서 글자 하나, 버튼 하나 바꾸는 것도 앱 심사를 다시 거쳐야 했다.

클라이언트 쪽에서는 그런 구분을 모르니까 "이거 수정해달라"고 하고 "언제 되냐"고 물어보는데, "심사 때문에 며칠 걸린다"는 말을 반복하는 게 스트레스였다.

---

## OTA 배포 발견

그러다 EAS Update, 즉 OTA(Over-the-Air) 배포를 알게 됐다.

앱 스토어 심사 없이 JS 번들만 교체하는 방식이다. 배포하면 앱이 백그라운드에서 새 번들을 받아두고, 다음 실행 때 적용된다. 글자 수정, UI 조정, 버그 픽스 같은 것들을 바로 반영할 수 있게 됐다.

---

## 두 번 껐다 켜야 하는 문제

한 가지 불편한 점이 있다. 업데이트가 있을 때 앱을 한 번 껐다 켜면 새 번들을 받고, 또 한 번 껐다 켜야 실제로 적용된다. 즉 **두 번 재시작해야 반영**된다.

이게 왜 그러냐면, 앱이 실행 중일 때 번들을 교체하면 현재 실행 상태와 충돌이 생길 수 있어서, 안전하게 "다음 실행 시 적용"하는 것이 기본 동작이기 때문이다.

실제 사용성 문제가 있냐고 하면 거의 없다. 일반 사용자는 업데이트가 있는지 모르는 채로 평소처럼 쓰다가 어느 순간 새 버전이 적용되어 있는 식이다. 강제로 팝업 띄우거나 사용을 막는 일이 없으니 사용자 입장에서 거슬릴 게 없다.

다만 "지금 당장 반영됐는지 확인해달라"는 요청에는 여전히 "앱 두 번 껐다 켜봐 주세요"라고 해야 한다는 점은 남아있다.

---

## 적용 범위 제한

OTA로 배포할 수 있는 건 **JS 코드 변경에 한정**된다. 네이티브 모듈을 새로 추가하거나, 네이티브 코드를 수정하는 경우에는 여전히 스토어 심사가 필요하다. UI 수정, 텍스트 변경, 비즈니스 로직 수정 등 대부분의 일상적인 수정은 OTA로 가능하다.

---

## 다음에는

> 네이티브 앱을 만든다면 OTA 배포는 프로젝트 시작할 때 무조건 세팅하고 들어가야 한다.
> 
> 
> 나중에 추가하는 것도 어렵지 않지만, 처음부터 있었다면 클라이언트 응대 방식 자체가 달라졌을 것이다.
> 
> "수정했습니다, 앱 껐다 켜보세요"가 가능한 것과, "심사 후 며칠 뒤에 됩니다"는 클라이언트 경험이 완전히 다르다.
>

# 모노레포 설계 회고 — AI에게 처음부터 지시했다면

---

## 실제로 어떻게 됐나

모노레포 구조를 잡고 시작했지만, 결과적으로 웹·앱이 각자 독립적으로 자랐다. 코드를 공유할 수 있는 구조는 갖춰져 있었는데, 처음부터 "공유할 것"을 명확히 지정하고 시작하지 않았다.

---

## 처음부터 지시했다면

### "타입은 한 군데서 관리해줘"

API 응답 타입, 도메인 모델(딜, 유저, 등급 등)을 공유 패키지에 정의하고, 웹과 앱이 거기서 가져다 쓰도록 해달라고 했을 것이다.

이렇게 시켰으면 API 응답 구조가 바뀔 때 웹·앱 중 한 곳만 고치고 다른 쪽을 놓치는 버그가 생기지 않았을 것이다. 타입이 안 맞으면 배포 전에 빌드 에러로 잡혔을 테니까.

### "가격 포맷, 등급 계산 같은 공통 함수는 공유 패키지에 짜줘"

날짜 포맷, 가격 표시, 등급 산정 로직처럼 플랫폼과 관계없는 함수들을 한 곳에만 두도록 했을 것이다.

이걸 지시하지 않았더니 같은 로직이 웹·앱에 각자 구현됐고, 포맷 규칙이 바뀌면 두 군데를 찾아서 고쳐야 했다.

### "API 호출 훅도 공유할 수 있으면 공유해줘"

웹과 앱이 같은 엔드포인트를 같은 방식으로 호출하는데, 각자 따로 짜는 것보다 한 번만 짜고 가져다 쓰는 게 낫다고 했을 것이다.

쿼리 키가 달라서 캐시가 따로 돌거나, 에러 처리 방식이 달라지거나 하는 일도 없었을 것이다.

### "웹·앱에서 둘 다 쓰는 UI 요소는 공유 가능하게 짜줘"

등급 뱃지, 사용자 아바타처럼 단순하고 양쪽에 다 나오는 컴포넌트는 처음부터 공유 패키지에 넣어달라고 했을 것이다.

React Native 기반으로 짜면 웹에서도 그대로 쓸 수 있는 구조이기 때문에, 처음부터 "이건 공유 컴포넌트로 만들어줘"라고 기준을 잡고 시작했으면 됐다.

---

## 왜 이렇게 안 됐나

처음에 "모노레포로 만들어줘"라고만 했지, 무엇을 공유할지를 지정하지 않았다. AI는 구조는 만들어줬지만, 공유 여부는 지시가 없으면 각자 앱 안에 짜는 쪽이 더 빠르고 안전하다. 결국 구조는 모노레포인데 실질적으로는 두 개의 독립 프로젝트처럼 됐다.

---

## 결론

> AI에게 모노레포를 지시할 때는 "구조를 만들어줘"로 끝내면 안 된다.
> 
> 
> **"어떤 것을 공유하게 만들어줘"까지 같이 지시해야** 의도한 대로 만들어진다.
>
