## 1. 관리자 시나리오

### TC-A01. 급식 상품 등록

**경로**: `/admin/meals/new`

| # | 행동 | 기대 결과 |
|---|------|----------|
| 1 | 관리자 로그인 → 사이드바 "급식 관리" 클릭 | `/admin/meals` 페이지 이동, 상품 목록 표시 (초기 빈 목록) |
| 2 | "상품 등록" 버튼 클릭 | `/admin/meals/new` 폼 표시 |
| 3 | 아래 정보 입력 후 "등록" 클릭 | 성공 토스트, `/admin/meals`로 이동, 목록에 새 상품 표시 |

**입력값**:

| 필드 | 값 |
|------|-----|
| 상품명 | `4월 중식` |
| 식사유형 | `중식(lunch)` |
| 가격 | `150000` (15만원) |
| 판매 시작일 | `2026-04-01` |
| 판매 종료일 | `2026-04-15` |
| 식사 시작일 | `2026-04-07` |
| 식사 종료일 | `2026-04-30` |
| 최대 인원 | `30` |
| 설명 | `4월 한 달 중식 급식입니다.` |

| # | 행동 | 기대 결과 |
|---|------|----------|
| 1 | 동일 절차로 석식 상품 등록 | 목록에 중식/석식 2개 상품 표시 |

**입력값**: 상품명 `4월 석식`, 식사유형 `석식(dinner)`, 가격 `120000`, 나머지 동일

### TC-A03. 일별 메뉴 입력

**경로**: `/admin/meals/{id}/menus`

| # | 행동 | 기대 결과 |
|---|------|----------|
| 1 | 상품 목록에서 "4월 중식" 클릭 → 상세 페이지 → "메뉴 관리" 이동 | 식사 기간(4/7~4/30) 내 날짜 목록 표시 |
| 2 | 4/7 날짜에 메뉴 입력: `쌀밥, 된장찌개, 제육볶음, 깍두기, 배추김치` → 저장 | 저장 성공 토스트, 해당 날짜에 메뉴 표시 |
| 3 | 4/8 날짜에 메뉴 입력: `쌀밥, 김치찌개, 고등어구이, 시금치나물, 배추김치` → 저장 | 저장 성공 |
| 4 | 4/7 메뉴 수정: `제육볶음` → `불고기`로 변경 후 저장 | 기존 메뉴 upsert되어 수정 반영 |
| 5 | 4/8 메뉴 삭제 | 삭제 성공, 해당 날짜 메뉴 빈칸으로 표시 |

### TC-S03. 급식 결제 (정상 플로우)

| # | 행동 | 기대 결과 |
|---|------|----------|
| 1 | "결제하기" 클릭 | `createMealOrder` 호출 → `meal_orders` INSERT (status: `pending`) |
| 2 | 결제 페이지(`/student/meals/pay/{orderId}`) 이동 | NICEPay 결제창(JS SDK `goPay`) 표시 |
| 3 | 테스트 카드 정보 입력 후 결제 진행 | NICEPay 인증 → `/api/payments/nicepay/confirm`으로 POST |
| 4 | confirm API 내부 검증 | ① `authResultCode` 확인 ② signature 위변조 검증 ③ 승인 API 호출 |
| 5 | 결제 성공 | `meal_orders.status = 'paid'`, `tid` 저장, `paid_at` 기록 |
| 6 | 결과 페이지(`/student/meals/pay/result`) | "결제 완료" 메시지 + 주문 정보 표시 |
| 7 | `payment_logs` 확인 | `action: 'approve'`, `status: 'success'` 기록 |

| 2 | 학생 | 신청 내역 확인 | 학부모가 결제한 건이 `student_id`로 조회됨 |
| 3 | 학생 | 동일 상품 결제 시도 | `이미 신청 중이거나 결제 완료된 주문이 있습니다.` |

### TC-X03. 학생 결제 → 관리자 강제취소 → 학생 확인

| # | 역할 | 행동 | 기대 결과 |
|---|------|------|----------|
| 1 | 학생 | 결제 완료 | `paid` |
| 2 | 관리자 | 강제 취소 (사유: `메뉴 변경으로 인한 전체 취소`) | NICEPay 취소 → `cancelled` |
| 3 | 학생 | 신청 내역에서 확인 | 상태 `cancelled`, 사유 표시 |
| 4 | 학생 | 동일 상품 재결제 시도 | 취소 후이므로 재결제 가능 |

### TC-X04. 판매 마감 시나리오

| # | 역할 | 행동 | 기대 결과 |
|---|------|------|----------|
| 1 | 관리자 | max_capacity=2 상품 등록 | - |
| 2 | 학생A | 결제 완료 | paid (1/2) |
| 3 | 학부모B (자녀B) | 결제 완료 | paid (2/2) |
| 4 | 학생C | 결제 시도 | `정원이 마감되었습니다.` |
| 5 | 관리자 | 학생A 강제 취소 | paid → cancelled (1/2) |
| 6 | 학생C | 다시 결제 시도 | 정원 여유 있으므로 성공 |

## 6. 데이터 검증 체크리스트

결제 완료 후 아래 데이터가 올바르게 기록되었는지 확인한다.

### meal_orders

| 컬럼 | 기대 값 |
|------|---------|
| `user_id` | 결제한 사용자 (학생 본인 또는 학부모) |
| `student_id` | 급식 대상 학생 |
| `product_id` | 선택한 급식 상품 |
| `order_id` | `MEAL-{uuid}` 형식 |
| `amount` | 상품 가격과 일치 |
| `status` | `paid` |
| `tid` | NICEPay TxTid 값 |
| `paid_at` | 결제 완료 시각 (UTC) |

### payment_logs

| 컬럼 | 기대 값 |
|------|---------|
| `order_type` | `meal` |
| `order_id` | `meal_orders.order_id`와 일치 |
| `action` | `approve` (결제), `cancel` (취소) |
| `status` | `success` |
| `result_code` | 카드 결제: `3001` / 취소: `2001` |
